1. Overview

EIR is a multi-agent clinical knowledge-graph extraction pipeline built for the UChicago AI hackathon on Clinical_KG_OS_LLM. Given a doctor–patient transcript, EIR produces a knowledge graph with seven entity types (SYMPTOM, DIAGNOSIS, TREATMENT, PROCEDURE, LOCATION, MEDICAL_HISTORY, LAB_RESULT) and six edge types (INDICATES, CAUSES, CONFIRMS, RULES_OUT, TAKEN_FOR, LOCATED_AT). The output is scored against the organizer's kg_similarity_scorer on a four-component composite metric.

The pipeline organizes thirteen specialist agents (one per entity type, one per edge type) in a two-stage parallel workflow. Each agent has its own prompt, model, sampling parameters, and reasoning toggle. Stage 1 fans out seven entity extractors that read the transcript and the curated label list; Stage 2 fans out six edge extractors that read the transcript and the entity list Stage 1 produced. The thirteen per-patient outputs are merged into one canonical KG via the organizer's BGE-M3 entity-resolution step (dump_graph.py).

2. Architecture: parallelization (sectioning) workflow

In Anthropic's "Building Effective Agents" taxonomy, EIR is the parallelization workflow with sectioning pattern: decompose a task into independent subtasks that run concurrently, then aggregate. We chain two stages of it.

┌─────────────────────────────────────────────┐ transcript ──► │ Stage 1 — fan out 7 entity specialists │ │ (SYMPTOM, DIAGNOSIS, TREATMENT, PROCEDURE, │ │ LOCATION, MEDICAL_HISTORY, LAB_RESULT) │ │ parallel · z-ai/glm-4.7-flash │ └────────────────────┬────────────────────────┘ │ entity list ▼ ┌─────────────────────────────────────────────┐ │ Stage 2 — fan out 6 edge specialists │ │ (INDICATES, CAUSES, CONFIRMS, RULES_OUT, │ │ TAKEN_FOR, LOCATED_AT) │ │ parallel · deepseek-r1-distill-qwen-32b │ └────────────────────┬────────────────────────┘ │ per-patient sub-KG ▼ ┌─────────────────────────────────────────────┐ │ Aggregation — convert → dump_graph │ │ (BGE-M3 ER @ 0.85 cosine) → unified KG │ └─────────────────────────────────────────────┘

The aggregation step (BGE-M3 entity resolution inside dump_graph.py) is the classic fan-in / merge that completes the parallelization pattern — bringing the thirteen parallel outputs back into one canonical KG.

4. Stage 1 vs Stage 2 — asymmetric model choice

Different cognitive tasks deserve different tools. Entity extraction is mostly retrieval + light classification (read transcript, identify clinical span, normalize); edge extraction is multi-step inference (decide whether a relation exists, disambiguate edge type, weigh competing differentials). We use a fast model for the high-volume retrieval task and a reasoning model only where the deliberation is needed.

Why z-ai/glm-4.7-flash for entities (Stage 1)

  • Single forward pass per category. No multi-step inference.
  • 140 calls per batch (7 categories × 20 patients), each emitting ~3k output tokens.
  • Reasoning-model pricing on this volume would dominate the budget for marginal quality gain.
  • Latency matters here because Stage 2 cannot start without Stage 1's entity lists.

Why deepseek/deepseek-r1-distill-qwen-32b for edges (Stage 2)

Edge extraction needs reasoning the fast model cannot do reliably:

  • Decide whether a clinical relation exists between two extracted entities.
  • Disambiguate edge type (a covid swab → covid-19 edge could be RULES_OUT or CONFIRMS depending on the doctor's framing).
  • Handle negation correctly (an absent X doesn't INDICATE a diagnosis but may RULES_OUT it).
  • Weigh competing differentials when one symptom maps to multiple candidate diagnoses.
  • Pick the right evidence quote across multiple candidate turns.

Empirically, fast models on edges produce edge-type confusion (RULES_OUT covid swabs misclassified as INDICATES) and weak src/dst pairings. Reasoning fixed both.

5. Why multi-agent fan-out, not transcript-based routing

Every transcript runs all 13 categories. We do not classify the transcript first and dispatch to a subset. Reasons:

  1. Routing requires reading the transcript first. A pre-pass LLM call per patient just to classify which agents to skip. Adds latency, another failure point, another cost.
  2. Recall safety. A router misclassification ("no labs in this transcript") permanently drops that category. With "run all", every LAB_RESULT call still gets a chance to find BP 148/90 mentioned in passing. Routing trades recall for cost; we chose recall.
  3. Empty categories are cheap. When a transcript truly has nothing for a category, the model returns empty buckets quickly. Saving ~4 such calls per patient × 20 patients ≈ 80 calls; the routing pre-pass costs ~20 calls. Net savings is small.
  4. Schema Completeness component (25% of organizer composite). Missing any of the 7 entity types or 6 edge types in the unified graph penalizes us. Always running all 13 guarantees a chance at each.
  5. Parallel fan-out. Within a transcript the 13 categories run concurrently; wall-clock is bounded by the slowest single category, not the sum. Routing to fewer doesn't shrink the critical path much.
  6. Deterministic shape. Every patient yields the same 13-file fan-out, which simplifies the converter, dump_graph, scoring, and any downstream tooling.
  7. Prompt specialization. Each of the 13 prompts is tuned to its category's semantics (denials in SYMPTOM, family-history naming in MEDICAL_HISTORY, src/dst type patterns in edges). One omnibus prompt can't carry that specialization without bloating to ~80k tokens.

6. Why split calls by category (13× more calls than a single omnibus prompt)

The split is intentional. It trades raw call count for quality, reliability, and modularity:

  1. Per-category prompt specialization. Each prompt is tuned to its category's semantics — SYMPTOM has denial rules, MEDICAL_HISTORY has family-history naming, edge prompts have src/dst type patterns and out-of-corpus anchors specific to that relation. One omnibus prompt would either drop these specializations or balloon past 80k tokens.
  2. Forced category coverage. With per-category prompts the model is explicitly asked to look for X. With one omnibus prompt, the model tends to extract whatever's most salient and skip the less obvious categories — populating the easy ones (SYMPTOM, DIAGNOSIS) and missing the rare ones (LAB_RESULT, CONFIRMS).
  3. Failure isolation. If the INDICATES call fails on a patient, only that one cell goes empty (1/120 edges). With one omnibus prompt, a single failure wipes out all 13 categories for that patient.
  4. Independent retry granularity. A 429 on TAKEN_FOR doesn't block CAUSES, RULES_OUT, etc. Each category retries on its own budget. With one big call, a single retry replays the entire 80k-token prompt.
  5. Parallel fan-out within a transcript. 13 categories run concurrently per transcript, wall-clock = max(category latency), not sum. One omnibus prompt is serial — one massive call must finish before any extraction is usable.
  6. Asymmetric model choice depends on the split. We can run entities on glm-4.7-flash and edges on deepseek-r1 only because the calls are separable. One omnibus prompt forces one model for everything.
  7. Cleaner output contracts. Each category has a focused JSON shape (matched / other / fine_grained with category-specific fields). One omnibus output would need a 13-section schema, harder to parse and validate.
  8. Iterability. When we needed to fix INDICATES, we edited only INDICATES_PROMPT and re-ran. With one omnibus prompt, every prompt edit risks regressing all 13 categories.
  9. Per-category telemetry. We can see "INDICATES hit% = 45.5%, TAKEN_FOR hit% = 73.3%" because calls are tagged by category.

Cost bound: the extra calls are absorbed by parallelism (wall-clock unchanged) and by the cheap-model-for-high-volume choice (Stage 1 at ~$0.10/M tokens, not the reasoning model). The 260 calls per batch don't cost meaningfully more than a single omnibus prompt would, because each is small (~3–4k input tokens).

7. Why we pass the whole transcript (not a summary)

  1. Verbatim quote requirement. Every emitted entity/edge must cite a turn_id and a quote taken verbatim from the transcript. A summary loses both — turn IDs disappear and exact phrasings get rewritten.
  2. Synonym bridging needs raw phrasing. Patient says "felt really hot" → curator wants subjective fever. The whole point of synonym mapping is to bridge lay phrasings to curated terms. A summary normalizes "felt really hot" to "fever" upstream — the bridge it's supposed to test is already collapsed.
  3. Denials require originals. "No fever" vs. "fever" vs. "I had a fever last year" — a summary often keeps one and drops the others, or fuses them into "denies fever". Our absent X denial form depends on seeing the raw P-N response in question context.
  4. Subjective/objective tie-breaks need the discriminator. pleuritic chest pain vs. bare chest pain is decided by whether the patient said "worse with deep breath". A summary like "patient has chest pain" drops the qualifier we need.
  5. Color priority, compound preservation, phonetic variants all operate on raw phrasings.
  6. No single summary serves 13 categories. Each prompt looks for different things in the same transcript. A pre-summarizer would have to preserve everything for everyone — at which point it's not a summary anymore.
  7. Information bottleneck. Summary is lossy and we can't undo it. A category's prompt may need a single phrase the summarizer dropped — and we'd never know.
  8. Cost is small. Transcripts are ~5–7k chars (~1.5k tokens). 13 calls × 1.5k = 20k tokens of transcript per patient. A summary call would still cost ~3k tokens to produce, with an extra failure point and quality loss. The saving doesn't exist.
  9. Reasoning model (Stage 2) needs the doctor's exact framing. deepseek-r1 disambiguates INDICATES vs. RULES_OUT vs. CONFIRMS by reading "we should rule out", "this is consistent with", "we'll do a swab to confirm". A summary normalizes those into "doctor considers diagnosis X" and the disambiguation signal is gone.
  10. Multi-turn evidence. Some edges require the symptom from one turn and the diagnostic framing from another. A summary may collapse those into a single line and lose turn-level provenance.

8. Why we did not adopt the other agentic patterns

Routing (transcript-based dispatch)

Already covered in §5 — pre-pass LLM call per patient, recall safety, Schema Completeness penalty, parallelism makes the cost savings tiny, deterministic shape simplifies downstream tooling.

Orchestrator-Workers (LLM decides decomposition at runtime)

  • The decomposition is fixed by the curator's ontology — 7 entity types, 6 edge types. Nothing for an orchestrator to decide.
  • An orchestrator adds another LLM call per transcript that doesn't produce final output, and introduces run-to-run variance.
  • Static decomposition already covers the full schema; the dynamic alternative buys nothing.

Evaluator-Optimizer (critic loop)

  • The natural evaluator is the organizer's kg_similarity_scorer, which sits downstream of extraction and uses a curator graph we don't have at runtime in the holdout setting. We can't call it from inside extraction.
  • LLM-as-judge alternatives would double the call count, add another failure mode, and still need quality criteria we'd have to define from scratch.
  • We optimize at prompt-engineering iteration time (v7 → v10), not per-call — cheaper, reproducible, auditable.

Reflection (self-critique)

  • Reflection at call level (extract → critique → re-extract) doubles latency and cost per call.
  • Our errors are mostly recall gaps (model missing things), not precision problems (model emitting wrong things). Reflection helps precision more than recall.
  • Single-pass approximations of reflection — the v9 "comprehensive review" / "recall floor" prompt rules — dropped precision without lifting recall enough to justify them. We removed them in v10.

ReAct / Tool use

  • The transcript is a self-contained input. No databases to query, no calculations to perform, no APIs to call during extraction.
  • Tool use would add infrastructure (tool definitions, execution layer, retry/fallback) for zero quality gain.
  • The reasoning model (deepseek-r1) does multi-step inference internally — that's the right place for deliberation, not externalized into tool calls.

Common thread: clinical KG extraction with a fixed ontology, a single text input, and a downstream similarity scorer is naturally a static parallel pipeline. The fancier patterns (orchestrator, evaluator, reflection, ReAct) add cost and variance that buy nothing in this domain — they're solutions to problems we don't have.

9. Three-bucket output schema

Every per-category extraction emits three buckets:

BucketMeaningWhen populated
matchedCurated label copied verbatim, anchored to a transcript turnWhen the transcript supports a label that's in the runtime <curated_labels> block
otherFree-text entity from the transcript that's NOT in the curated listWhen the model spots a clinical entity that the curator's vocabulary doesn't cover
fine_grainedA more specific subtype of a curated labelWhen the transcript provides a refinement (qualifier, dose, severity) finer than what's in the curated list

After extraction, the converter (convert_v7_to_kg_extraction_format_260425.py) flattens all three buckets into one flat nodes[] array per patient. The bucket distinction is invisible to dump_graph and to the scorer — it only exists at extraction time as a way to keep precision (other / fine_grained) and recall (matched) on separate dials.

No-corpus consequence. If the <curated_labels> block is empty (because the holdout transcripts don't ship with a curator label list), matched is logically empty (no list to match against) and fine_grained is also empty (no reference for "finer than"). Everything legitimately lands in other. The scorer's curator-alignment still works because the converter flattens the buckets. But the side effect is that Stage 1 emits more entities into other when there's no curated anchor, which makes Stage 2 edge prompts larger (~15k chars vs. ~11k) and amplifies upstream rate-limit pressure.

Building your own KB

Any unified graph from a previous pipeline run can become a KB seed for the next iteration. The merge logic is a simple deduplication by (label, entity_type): 

seen = set()
combined = []
for source_kb in [original_222_kb, your_new_kb]:
    for entry in source_kb:
        key = (entry["label"].strip().lower(), entry["entity_type"].upper())
        if key in seen: continue
        seen.add(key)
        combined.append(entry)
json.dump(combined, open("your_new_combined_kb.json", "w"), indent=2)

12. Convert → dump_graph → scorer chain

The wrappers auto-run the post-batch chain:

Step 1 — Convert

convert_v7_to_kg_extraction_format_260425.py takes our per-category three-bucket output (one JSON per (patient, category) = 260 files per batch) and produces one per-patient JSON in the format the organizer's dump_graph.py consumes: 

{
  "nodes": [
    {"id": "N_001", "text": "...", "type": "SYMPTOM", "evidence": "...", "turn_id": "P-1"},
    ...
  ],
  "edges": [
    {"source_id": "N_001", "target_id": "N_005", "type": "INDICATES", "evidence": "...", "turn_id": "D-2"},
    ...
  ]
}

Buckets are flattened (matched + other + fine_grained → one nodes[] list). Edge endpoints are resolved by matching (text, allowed_type) against Stage 1 nodes; unresolved endpoints (e.g., model-emitted entity text that Stage 1 didn't produce) are dropped and counted in the converter's summary.

Step 2 — dump_graph.py (organizer's tool, BGE-M3 entity resolution)

Reads all per-patient JSONs in the input directory; runs BGE-M3 cosine similarity at threshold 0.85 within each entity type to merge near-duplicate canonical labels (e.g., "high blood pressure" ↔ "hypertension"). Outputs one unified KG with canonical entity nodes and resolved edges, tagged with each occurrence's (res_id, turn_id).

Step 3 — kg_similarity_scorer.py (organizer's scorer)

Compares our unified graph to the curator baseline using fuzzy string match (SequenceMatcher ≥ 0.80) within each entity type. Computes a four-component composite score (see §14).

13. The out-of-corpus (OoC) eval bundle

To stress-test whether the pipeline generalizes beyond the in-corpus respiratory vocabulary, we built an out-of-corpus evaluation set from the ACI-Bench dataset (mkieffer/ACI-Bench).

What's in eir_aci_bench/

  • 20 transcripts from ACI-Bench's train split, converted from [doctor]/[patient] tags to our D-N/P-N turn-ID format (see pull_aci_bench_260425.py). Spans cardiology, endocrinology, GI, ortho, derm — vocabulary largely disjoint from the in-corpus 20.
  • 20 paired clinical notes (the ACI-Bench gold-standard summaries).
  • 20 hand-curated KGs built by Claude (Anthropic), one per encounter, in the same schema as the organizer's RES0XXX_curated_kg.json. Total: 558 nodes, 261 edges.
  • unified_graph_curated_aci.json: the 20 curator KGs merged via dump_graph.py (BGE-M3 ER @ 0.85 cosine) → 338 canonical nodes, 261 edges. This is the OoC scoring baseline.
  • er_merge_decisions_curated_aci.json: provenance of which surface variants were merged into which canonical labels.

When you run ./run_v10_260426.sh aci-bench, the wrapper points --baseline at unified_graph_curated_aci.json automatically.

14. Score history and what each component means

The composite score

The organizer's scorer produces a weighted four-component composite:

ComponentWeightWhat it measures
Entity F125%Harmonic mean of node-level precision and recall, fuzzy-matched against the curator baseline at SequenceMatcher ≥ 0.80, scoped within entity type. The most discriminating component on this dataset.
Population Completeness25%Per-patient coverage: of each curator patient's nodes, how many do we cover? Averaged across patients.
Relation Completeness25%"At least one of each edge type exists in the unified graph." Saturates to 1.0 trivially at 20-patient scale.
Schema Completeness25%"At least one of each entity type exists in the unified graph." Same — saturates to 1.0 trivially.

Note: Relation and Schema components are designed for very small KGs to avoid degenerate cases. At our scale they always max out, so the meaningful signal comes from Entity F1 and Population.

The combined-KB row tests the hypothesis that the curator-vocabulary anchor in the prompt is the active ingredient: extending the KB to cover the OoC vocabulary recovers some of the recall lost on the prior OoC run (+2.4 pp recall over the 222-label KB).

The deterministic BGE-M3 canonicalization post-step (see §15) rewrites each student node's text to the nearest curator label of the same entity type when the cosine similarity is ≥ 0.70. This bridges lay-language → clinical-vocabulary surface variants that the scorer's lexical fuzzy match (SequenceMatcher ≥ 0.80) misses. Net OoC lift over the unbridged combined-KB run: +4.0 pp composite, +15.9 pp Entity F1, +13.4 pp Recall.

The four targeted prompt edits (260427) are: (1) SYMPTOM physician-exam-findings rule for clinician-observed signs (bruits, tenderness, murmurs, fluctuance); (2) SYMPTOM stacked-ROS rule for compound denial questions ("any fever, chills, cough?" → emit three separate absent X); (3) MEDICAL_HISTORY activity-context + prior-procedures rules (yard work, prior knee arthroplasty, hypertension history); (4) LAB_RESULT outcome-anchor disambiguation (chest x-ray unremarkable is one LAB_RESULT, not two entities) and PROCEDURE bare-test-name mirror rule. Combined with canonicalization, these lift OoC composite from 0.864 → 0.897 (+3.3 pp), Entity F1 from 0.612 → 0.633 (+2.1 pp), and Recall from 60.9% → 68.1% (+7.2 pp). Population jumps to 0.955 because the new rules surface exam-finding and prior-procedure nodes the model previously dropped.

The "loose-other" in-corpus run additionally relaxes "When in doubt, emit nothing" → "Emit when clearly clinical" across MEDICAL_HISTORY, TREATMENT, PROCEDURE, and LAB_RESULT (SYMPTOM and LOCATION stay tight to protect precision; per-category trade-off in §15b). This pushes Population to 1.0 in-corpus and is the configuration the submission ships with by default.

Per-prompt tight-vs-loose calibration

The 7 entity prompts are not all calibrated the same way. SYMPTOM and LOCATION extract too many false positives at high recall (long tail of clinical-sounding transcript chatter), so their other / fine_grained rules stay tight ("when in doubt, emit nothing"). The other four are loose ("emit when clearly clinical") to push the Population component to 1.0. Empirical evidence: loose-mode SYMPTOM added 44 unmatched student nodes in-corpus and dropped precision −22.9 pp without lifting recall.

PromptModeWhy
SYMPTOMTIGHTLoose mode = +44 unmatched nodes, −22.9 pp precision; recall barely moved
LOCATIONTIGHTBounded vocabulary; loose adds geographic / institutional false positives
DIAGNOSISNEUTRALAlready 84% recall; no emit-nothing rule originally
TREATMENTLOOSELifestyle/avoidance instructions and drug-class names benefit from inclusion
PROCEDURELOOSEBare test/imaging names (creatinine, lumbar mri) often missed under tight rules
LAB_RESULTLOOSEOutcome-anchored compounds (chest x-ray unremarkable) need free emission
MEDICAL_HISTORYLOOSEActivity context (yard work, tennis exertion) and prior-procedure framing have long tails

The _entity_prompt helper (used for PROCEDURE, LOCATION, LAB_RESULT) accepts tight=True to flip its rules. LOCATION_PROMPT sets it. SYMPTOM and MEDICAL_HISTORY have their other / fine_grained rules edited inline.

15. The BGE-M3 canonicalization post-step

After dump_graph builds the unified graph but before the scorer compares it to the curator baseline, we run a deterministic post-step: canonicalize_to_curator_260426.py. For each student node it embeds the node text with BGE-M3, finds the highest-cosine curator label of the same entity type, and — if cosine ≥ 0.70 — rewrites the node's text field to that label. The original text is preserved in text_original and the cosine score in canonicalized_via_bge_cosine for full auditability. Edges follow nodes for free because edges reference source_id / target_id, not text.

Why this exists

The scorer's fuzzy match (SequenceMatcher ≥ 0.80) catches lexical near-duplicates but misses lay → clinical bridges that are semantically obvious. Examples that BGE-M3 catches at threshold 0.70 but SequenceMatcher misses:

  • swelling in my ankles → ankle swelling (cosine 0.934)
  • blood pressure 156 over 94 → blood pressure 156/94 (0.933)
  • zoloft, um, 25 mg once a day → zoloft (0.705)
  • cholesterol was like, a tiny bit high → elevated cholesterol (0.700)
  • insect bite → tick bite (0.800)
  • heart failure → congestive heart failure (0.803)

What it changed in the OoC run

Of 428 student nodes scanned, 144 (33.6%) were rewritten to a canonical curator label. By type: LOCATION 27, TREATMENT 24, SYMPTOM 24, LAB_RESULT 23, DIAGNOSIS 16, MEDICAL_HISTORY 16, PROCEDURE 14.

Threshold sweep

We tested two thresholds on the same OoC unified graph:

ThresholdRewritesCompositeEntity F1PrecisionRecallF1
none (no canon)00.8240.45343.4%47.5%45.3%

0.80 is more conservative — it caught only the high-confidence near-duplicates and gave back about two-thirds of the gain. 0.70 caught two questionable boundary rewrites (using the knee → exercise, bowels → stomach) but the net effect is strongly positive because they are minor and the dozens of correct rewrites at 0.70–0.80 outweigh them. We ship with 0.70.

Why this is deterministic, not an Agent 5

This step is a pure embedding-cosine lookup with no LLM call. That keeps the pipeline's run-to-run variance contained to the two LLM stages (Stage 1 entity extraction, Stage 2 edge inference) and makes the canonicalization auditable bit-for-bit: the same input graph + same KB + same threshold always produces the same output graph.

How to override the threshold

python canonicalize_to_curator_260426.py \
    --unified  path/to/unified_graph_*.json \
    --kb       curated_synthetic_kb_combined_260426.json \
    --output   path/to/unified_graph_*_canonicalized.json \
    --report   path/to/canonicalize_report.json \
    --threshold 0.70

17. Limitations and lessons learned

Model choice trades Entity F1 quality against Population count

We A/B-tested the default pairing (z-ai/glm-4.7-flash Stage 1 + deepseek/deepseek-r1-distill-qwen-32b Stage 2) against the strongest alternative pairing the published benchmarks supported (openai/gpt-oss-20b Stage 1 + nvidia/nemotron-3-nano-30b-a3b Stage 2). The first attempt silently failed to swap Stage 1 because the OpenRouter client's __init__ default value model=OPENROUTER_MODEL was captured at class-definition time, so monkey-patching the module constant alone left the client constructed with the stale default. We fixed the wrapper to also patch v8.OpenRouterClient.__init__.__defaults__ in-place. After the fix, every per-call log line for the swap run shows the correct model strings, with zero z-ai/glm-4.7-flash calls. Same in-corpus 20 transcripts, same 522-label KB, same prompts, same canonicalization @ 0.70.

ConfigurationCompositeF1PrecisionRecallPopulationNodes
Default models, run A (260427_005525)0.9250.69958.5%86.9%1.000374
Default models, run B (260427_013403)0.9290.71561.4%85.5%1.000345
REAL swap (gpt-oss + nemotron) (260427_082458)0.8910.80377.9%82.8%0.763254

The model swap moves the score significantly — but in opposing directions on different components. gpt-oss-20b is dramatically more precise (RLHF discipline confirmed: precision +16.5 pp, recall −2.7 pp, Entity F1 +8.8 pp — the highest in-corpus F1 we have ever recorded, 0.803). But it is also more conservative: it emits 254 student nodes vs 345 for the default — 91 fewer. Population is a count-based metric (min(student_count / baseline_count, 1.5) / 1.5) that punishes this conservatism: Population drops from 1.000 to 0.763. Since each component is weighted 25%, the +8.8 pp F1 lift contributes +0.022 to composite, while the −0.237 Population drop contributes −0.059 — net composite −0.038.

The original (pre-fix) A/B numbers were misleading because Stage 1 never changed; only Stage 2 swapped to nemotron. Stage 2 cannot move composite by construction (Relation Completeness is already saturated at 1.0 with our 6-edge fan-out, so edge model swaps only affect edge_jaccard which is reported but not weighted into composite). The earlier conclusion that "model choice does not move the score" was wrong; it was based on a partial swap.

Implications

The KG produced by gpt-oss-20b has higher F1 (0.803) and 16.5 pp more precision than the glm-4.7-flash default — it is clinically more defensible. But the composite scorer rewards quantity (Population) at the same weight as quality (F1), so the smaller-but-better KG scores 0.038 composite lower. Two reasonable routes from here:

  • Composite-optimized: keep z-ai/glm-4.7-flash for Stage 1. Optionally switch Stage 2 to nvidia/nemotron-3-nano-30b-a3b for 2× speed at equal score (Stage 2 cannot move composite).
  • F1-optimized (recommended for downstream QA): switch to openai/gpt-oss-20b Stage 1, then aggressively loosen other / fine_grained emission rules across more categories (currently SYMPTOM and LOCATION are TIGHT) to push student node count from 254 → 333+ and saturate Population. This should yield both the F1 (~0.80) and Population (~1.0) wins, with composite ≈ 0.94+.

Curator-anchor dependency

v10's in-corpus performance (0.930 composite) depends on the <curated_labels> block being populated with the curator's specific phrasings. When we ran v10 with an empty curated-labels block (zero-shot mode), composite dropped to 0.815 with Entity F1 of 0.262 — a fundamental degradation. The pipeline needs either a curator-supplied label list at runtime or a substitute vocabulary (SNOMED-CT subset, UMLS, our combined KB).

OoC vocabulary mismatch is the dominant failure mode (now mitigated)

On out-of-corpus transcripts where the curator's vocabulary doesn't cover the active conditions (CHF, anemia, kidney stones, etc.), recall drops sharply because the model's natural extractions don't match the curator's specific surface forms. We mitigate this with two layers: (a) an extended KB that covers the OoC vocabulary (the combined 522-label KB), and (b) a deterministic BGE-M3 canonicalization post-step at cosine ≥ 0.70 (see §15). Together these lift OoC composite from 0.815 → 0.864 and Entity F1 from 0.440 → 0.612.

LLM nondeterminism

Temperature 0.6 on deepseek-r1-distill produces ~9 pp swings in INDICATES hit% between same-prompt runs. Single-sample comparisons of prompt edits are unreliable; multi-sample averaging (3–5 runs) is needed for confident A/B comparisons.

Recall-pressure prompt rules backfired

Earlier iterations (v9) added explicit "recall floor" / "comprehensive review" / "matched=RECALL-ORIENTED" framing to push the model toward higher emission rates. This dropped precision sharply (76.9% → 59.6%) without lifting recall enough to justify it. We removed those levers in v10. Lesson: instruction-tuned models are sensitive to over-specification; cleaner prompts often outperform feature-rich ones.

What we did not have time to try

  • An LLM-based canonicalization agent (Agent 5 in a 5-stage pipeline) that uses generative reasoning to map free-text to canonical clinical terms via SNOMED-CT or UMLS. We shipped a deterministic BGE-M3 cosine version instead (see §15), which gave us +15.9 pp Entity F1 with no LLM cost or nondeterminism. An LLM-based variant could potentially go further on the harder bridges still missed at cosine 0.70.
  • Two-pass recall sweep (script-side, not prompt-side): for every curator label not yet emitted, force a second-pass focused look.
  • Multi-sample LLM averaging at temperature 0.6 to dampen nondeterminism.
  • Hard wall-clock cap per LLM call (some Stage 2 calls binge for 20+ minutes; an 8-min cap with one refire would bound batch tail latency without changing extraction quality).
