## Quick start

```bash
# 1. Install deps (use a venv — llama_index + torch are heavy)
pip install -r requirements_260426.txt

# 2. Add your OpenRouter API key
echo "OPENROUTER_API_KEY=sk-or-v1-..." >> .env

# 3. Run v10 in-corpus, scored against the in-corpus curator
./run_v10_260426.sh in-corpus

# 4. Run v10 on the OoC ACI-Bench transcripts, scored against our OoC baseline
./run_v10_260426.sh aci-bench
```

## Knobs the team can change without editing code

### `EIR_CURATED_KB_PATH` — swap the curated KB at runtime

Default (set by `run_v10_260426.sh`): `curated_synthetic_kb_combined_260426.json`.

To use only the in-corpus 222 labels:
```bash
EIR_CURATED_KB_PATH=$(pwd)/curated_kb_260419.json \
  python smoke_test_v10_260425.py
```

To use your own KB:
```bash
EIR_CURATED_KB_PATH=$(pwd)/your_custom_kb.json \
  python smoke_test_v10_260425.py
```

### Custom KB schema

Each entry in the JSON list must have:
```json
{
  "entity_id":   "any_unique_string",
  "label":       "the canonical label (matches what curators write)",
  "entity_type": "SYMPTOM | DIAGNOSIS | TREATMENT | PROCEDURE | LOCATION | MEDICAL_HISTORY | LAB_RESULT",
  "description": "free text shown to the model",
  "aliases":     [],
  "source":      "human_curated | aci_bench_synthetic | ...",
  "source_code": "anything"
}
```

To build a new KB from any pipeline run's unified graph: copy the merge logic from
the cell that produced `curated_synthetic_kb_combined_260426.json` (see
`eir_implementation_log.md` doc-scope section).

### `--node-model MODEL` / `--edge-model MODEL` — swap models at runtime

Override the Stage 1 (entity extraction) and Stage 2 (edge / relation
reasoning) models on the command line. No code edits required.

Defaults:
- Stage 1: `z-ai/glm-4.7-flash`
- Stage 2: `deepseek/deepseek-r1-distill-qwen-32b`

Allowed OpenRouter models (per the organizer's README):

| Model | Notes |
|---|---|
| `z-ai/glm-4.7-flash` | Fast / SOTA tool calling (default Stage 1) |
| `qwen/qwen3-14b` | Smallest, fastest, dense |
| `nvidia/nemotron-3-nano-30b-a3b` | MoE, top reasoning, 2.2× faster than gpt-oss |
| `openai/gpt-oss-20b` | MoE, highest documented HealthBench (medical knowledge) |
| `deepseek/deepseek-r1-distill-qwen-32b` | Heavy CoT reasoning (default Stage 2) |

Examples:
```bash
# Try gpt-oss for entities, nemotron for edges
./run_v10_260426.sh in-corpus \
    --node-model openai/gpt-oss-20b \
    --edge-model nvidia/nemotron-3-nano-30b-a3b

# Try qwen3-14b for entities, keep default reasoning model
./run_v10_260426.sh aci-bench --node-model qwen/qwen3-14b
```

The flags monkey-patch `v8.OPENROUTER_MODEL` (Stage 1) and rebuild
`v8.MODEL_OVERRIDES_BY_CATEGORY` (Stage 2) at wrapper-import time, before
any worker thread runs.

### `--workers N`, `--res-ids ...`, `--no-score`, `--baseline PATH`

Standard CLI flags pass through to `v9.main()`. See each wrapper's docstring
or run with `--help`.

### `--no-canonicalize`, `--canon-threshold T` — control the BGE-M3 canon step

`--no-canonicalize` skips the post-step entirely; `--canon-threshold 0.75`
tightens the cosine threshold (default 0.70). See `canonicalize_to_curator_260426.py`
for what the post-step does.

## Architecture summary

**Pattern**: parallelization workflow with sectioning (Anthropic's "Building
Effective Agents" taxonomy). Two stages, each fanning out to specialist agents
that run concurrently.

```
                     ┌─────────────────────────────────────────────┐
   transcript ──►    │  Stage 1 — fan out 7 entity specialists     │
                     │  (SYMPTOM, DIAGNOSIS, TREATMENT, PROCEDURE, │
                     │   LOCATION, MEDICAL_HISTORY, LAB_RESULT)    │
                     │  parallel, glm-4.7-flash                    │
                     └────────────────────┬────────────────────────┘
                                          │ entity list
                                          ▼
                     ┌─────────────────────────────────────────────┐
                     │  Stage 2 — fan out 6 edge specialists       │
                     │  (INDICATES, CAUSES, CONFIRMS, RULES_OUT,   │
                     │   TAKEN_FOR, LOCATED_AT)                    │
                     │  parallel, deepseek-r1 (reasoning)          │
                     └────────────────────┬────────────────────────┘
                                          │ per-patient sub-KG
                                          ▼
                     ┌─────────────────────────────────────────────┐
                     │  Aggregation — convert → dump_graph         │
                     │  (BGE-M3 ER @ 0.85 cosine) → unified KG     │
                     └─────────────────────────────────────────────┘
```

Why this architecture (not orchestrator / reflection / ReAct / tool use): see
`eir_implementation_log.md`.

**OoC headline (submission default config): composite 0.897, Entity F1 0.633, Recall 68.1%.**
**Highest in-corpus Entity F1 we have recorded: 0.803, with `openai/gpt-oss-20b` on Stage 1.** This run scored 0.891 composite — −0.038 vs the glm-4.7-flash default — because gpt-oss-20b's RLHF discipline reduces total node count from 345 → 254, dropping Population (a count-based metric) from 1.000 → 0.763. The composite punishes the smaller-but-cleaner KG even though Entity F1 went up +8.8 pp. See the "Model A/B test" section in `eir_implementation_log.md` for the full analysis.

The "4 prompt edits" are: SYMPTOM physician-exam-findings + stacked-ROS denials, MEDICAL_HISTORY activity-context + prior-procedures, LAB_RESULT outcome-anchor disambiguation, PROCEDURE bare-test-name disambiguation. The "loose-other" config additionally relaxes "When in doubt, emit nothing" → "Emit when clearly clinical" across MEDICAL_HISTORY, TREATMENT, PROCEDURE, LAB_RESULT (SYMPTOM and LOCATION stay tight to protect precision — see Prompt configuration matrix below).

## Prompt configuration matrix (260427)

The 7 entity prompts are not all calibrated the same way. SYMPTOM and LOCATION
extract too many false positives at high recall (long tail of clinical-sounding
transcript chatter), so their `other` / `fine_grained` rules stay tight.
The other four are looser to push Population component to 1.0.

| Prompt | `other`/`fine_grained` mode | Why |
|---|---|---|
| **SYMPTOM** | **TIGHT** ("when in doubt, emit nothing") | Loose mode added 44 unmatched student nodes, dropped precision −22.9 pp without lifting recall |
| **LOCATION** | **TIGHT** | Bounded vocabulary; loose mode adds geographic/institutional false positives |
| DIAGNOSIS | NEUTRAL (no emit-nothing rule) | Already low miss rate (16%); no tightening needed |
| TREATMENT | LOOSE ("do NOT under-emit") | Lifestyle/avoidance instructions and drug-class names benefit from being included |
| PROCEDURE | LOOSE | Bare test/imaging names (creatinine, lumbar mri) often go uncaptured under tight rules |
| LAB_RESULT | LOOSE | Outcome-anchored compounds (`chest x-ray unremarkable`) improve recall when freely emitted |
| MEDICAL_HISTORY | LOOSE | Activity context (yard work, tennis exertion) and prior-procedure framing have long tails |

The `_entity_prompt` helper (used for PROCEDURE / LOCATION / LAB_RESULT)
accepts `tight=True` to flip its rules; LOCATION_PROMPT sets it. Other prompts
have their `other` / `fine_grained` rules edited inline.

## Recent script-level fixes

- **`convert_v7_to_kg_extraction_format_260425.py`** — sanitizes the output
  filename by stripping internal underscores from the `res_id`. Without this,
  ACI-Bench patient IDs like `RES_D2N001` collapsed to `res_id="RES"` when
  `dump_graph` parsed them via `stem.split("_")[0]`, merging all 20 patients
  into one bucket (Per-Patient Coverage 0.00%).

- **`run_v10_260426.sh`** — resolves PROJECT_ROOT and EIR_ROOT from the script
  location and `cd`s to PROJECT_ROOT before invoking the wrappers, so v8 path
  resolution finds `data/transcripts/` and the human-curated baseline whether
  the script is invoked from `eir/EIR_260426/` (dev) or from a sibling path.
  Also auto-symlinks `eir/.env` → `<project_root>/.env` if the wrapper-side
  path is missing.

- **`run_postchain_260426.sh`** (new) — operates on an existing extraction run
  dir to (re)run convert / dump_graph / canonicalize / score. Useful when an
  in-line run hit a transient API failure mid-chain or you want to re-score
  with a different KB or canon threshold without re-extracting.

- **`canonicalize_to_curator_260426.py`** — deterministic BGE-M3 cosine post
  step (≥ 0.70 by default) that rewrites each student node's `text` to the
  nearest curator KB label of the same `entity_type`. Bridges lay → clinical
  surface variants the scorer's lexical fuzzy match (≥ 0.80) misses. Wired
  between `dump_graph` and `kg_similarity_scorer` in both wrappers; skip with
  `--no-canonicalize`. Net OoC lift: +15.9 pp Entity F1, +0.04 composite.
